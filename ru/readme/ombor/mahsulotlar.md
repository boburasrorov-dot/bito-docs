# Товары

<details>

<summary>Создание товаров</summary>

Через раздел Товары осуществляется процесс создания, редактирования и управления всеми товарами на складе. Этот раздел служит для хранения основной информации о товарах и связи их со складом, продажами и другими процессами.

Пользователь заходит из левого меню в раздел **Склад → Товары** и нажимает кнопку **"Создать"** для добавления нового товара.

<figure><img src="../../.gitbook/assets/image (371).png" alt=""><figcaption></figcaption></figure>

При создании товара вводятся следующие данные:

* **Название** — вводится название товара
* **Категория** — выбирается категория товара
* **Единица измерения** — тип измерения товара (штука, кг, литр и другие)
* **Тип упаковки** — выбирается тип упаковки товара
* **Количество в упаковке** — количество товара в одной упаковке
* **Штрих-код упаковки** — отдельный штрих-код для упаковки
* **SKU** — уникальный идентификационный код товара
* **Штрих-код** — основной штрих-код товара
* **Поставщик** — поставщик товара
* **Место на складе** — расположение товара (стеллаж, сектор и другие)
* **Примечание** — дополнительная информация (необязательно)
* **Загрузка файла** — прикрепление файла, относящегося к товару

Настройки инвентаризации:

* **Товар / Полуфабрикат / Сырье** — указывается тип товара, какой выбран, тем и считается товар.
* **Вариантный товар** — использование в качестве вариантного товара
* **Маркированный товар** — для специальной маркировки
* **Срок годности** — ведение срока годности

<figure><img src="../../.gitbook/assets/image (373).png" alt=""><figcaption></figcaption></figure>

Состав товара (дополнительные настройки):

В этом разделе можно настроить параметры товара, связанные со складом и продажами:

* **Организация** — к каким организациям относится товар
* **Максимальное количество** — предел максимального запаса
* **Нормативные дни** — если товар закончится и будет оформлен заказ, через сколько дней этот товар поступит (для аналитики)
* **Желтая линия** — уровень предупреждения
* **Красная линия** — критический минимальный уровень
* **Доступен в продаже** — выпуск товара в продажу
* **Розничная** — розничная цена
* **Оптовая** — оптовая цена
* **Другие цены пользователь создает самостоятельно**

Через этот раздел товары полностью управляются и служат основой для последующих процессов (приход, расход, инвентаризация, продажи).

</details>

<summary>Создание вариантного товара</summary>

Если товар доступен в нескольких вариантах (например, цвет, размер и другие), их можно управлять не как отдельные товары, а через варианты внутри одного товара.

Для этого в окне создания товара включается переключатель **"Вариантный товар"**. При включении этой функции в системе открывается дополнительное окно **"Вариантный товар"**.

<figure><img src="../../.gitbook/assets/image (375).png" alt=""><figcaption></figcaption></figure>

В этом разделе варианты товара вводятся следующим образом:

* **Вариант** — выбирается тип варианта товара (например: цвет, размер)
* **Атрибут** — вводятся значения, относящиеся к выбранному варианту (например: белый, черный)

После ввода вариантов нажимается кнопка **"Добавить товары"**. После этого система автоматически формирует раздел **"Состав товара"**.

<figure><img src="../../.gitbook/assets/image (376).png" alt=""><figcaption></figcaption></figure>



В разделе состава товара:

* для каждого варианта создается отдельный товар (SKU)
* **SKU** — вводится отдельно для каждого варианта или
* генерируется системой через **Автоматическое создание**

В результате товары автоматически умножаются в соответствии с количеством введенных атрибутов.

Например:\
Если для товара "Телефон" выбран вариант **цвет**, и в качестве атрибута введены **красный** и **черный**, система автоматически создаст 2 товара:

* Телефон / красный
* Телефон / черный

Этот подход позволяет системно управлять товарами, учитывать каждый вариант отдельно и упрощать торговые процессы.

<figure><img src="../../.gitbook/assets/image (377).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (379).png" alt=""><figcaption></figcaption></figure>

Создание товара через варианты позволяет более структурно и эффективно управлять товарами в системе.

Использование этой функции дает следующие преимущества:

* **Упорядоченное управление товарами** — различные варианты одного товара (цвет, размер и другие) хранятся внутри одного товара, система не запутывается
* **Отдельный контроль каждого варианта** — для каждого варианта можно вести отдельный SKU, цену и остатки
* **Уточнение складского учета** — например, товары "красный" и "черный" учитываются отдельно, четко видно, какого больше или меньше
* **Упрощение торгового процесса** — во время продажи выбирается точный вариант, ошибки уменьшаются
* **Автоматическое создание товаров** — на основе атрибутов вариантов система автоматически создает несколько товаров, нет необходимости создавать вручную
* **Гибкость** — легко добавлять или изменять новые варианты, удобно расширять систему

В результате управление товарами через варианты выходит на более профессиональный уровень, а складские и торговые процессы работают намного точнее и быстрее.

</details>

<summary>Создание маркированного товара</summary>

Если при создании товара включается переключатель **"Маркированный товар"**, для этого товара каждая единица ведется с отдельной **уникальной идентификацией (маркой)**.

В этой функции товар управляется не по простому количеству, а **в разрезе каждой штуки**.

<figure><img src="../../.gitbook/assets/image (381).png" alt=""><figcaption></figcaption></figure>

В результате в системе осуществляется следующее:

* для каждого товара прикрепляется отдельный **уникальный код (серийный номер, IMEI и другие)**
* движение товара (приход, расход, инвентаризация, продажа) отслеживается именно по этой марке
* одинаковые товары не смешиваются, каждый контролируется отдельно

Как работает процесс:

При инвентаризации, приходе или продаже:

* система требует ввода не количества товара, а **отдельной марки для каждой штуки**
* пользователь вводит уникальные коды для каждого товара вручную или загружает их
* система сохраняет эти марки и идентифицирует именно этот товар в последующих операциях



При инвентаризации:

<figure><img src="../../.gitbook/assets/image (380).png" alt=""><figcaption></figcaption></figure>



В продаже:

<figure><img src="../../.gitbook/assets/image (383).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (385).png" alt=""><figcaption></figcaption></figure>

Например:\
Если товар "iPhone" создан как маркированный товар, при приходе на склад 10 штук вводится отдельный номер IMEI для каждого iPhone. Впоследствии при продаже или расходе в системе точно видно, какой именно IMEI был продан.

Польза:

* точный контроль дорогих и важных товаров
* предотвращение потерь и путаницы
* легкое ведение гарантийных и сервисных процессов
* отслеживание истории каждого товара (от кого поступил, когда продан)

Этот функционал обычно применяется **для магазинов техники, торговли электроникой и товаров, работающих с серийными номерами**.

<details>

<summary>Создание товара со сроком годности</summary>

<figure><img src="../../.gitbook/assets/image (387).png" alt=""><figcaption></figcaption></figure>

Если при создании товара включается переключатель **"Срок годности"**, для этого товара включается работа со сроком годности (expiry date).

Это означает, что товар управляется не по простому количеству, а **в разрезе срока годности**.

Этот функционал в основном применяется для продуктов питания, медикаментов и быстро портящихся товаров.

Как работает процесс:

* **В процессе закупки (прихода)**\
  при приеме товара на склад для каждого товара **вводится срок годности**
* **При хранении на складе**\
  товары хранятся в отдельных партиях по сроку годности
* **В торговом процессе**\
  при продаже товара пользователь выбирает, **товар с каким сроком годности** он продает

<figure><img src="../../.gitbook/assets/image (388).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (389).png" alt=""><figcaption></figcaption></figure>

В результате в системе обеспечивается следующее:

* товары точно контролируются по сроку
* появляется возможность отслеживать товары с приближающимся сроком
* уменьшаются ошибки при работе с просроченными или истекшими товарами
* облегчается применение принципа FIFO (первым пришел — первым продан)

Например:\
Если для товара "Кефир" включен срок годности, при приходе на склад для каждой партии вводится дата (например: 23.03.2026). При продаже же товар выбирается и продается именно по этой дате.

Это играет важную роль в контроле качества товара и снижении потерь.







</details>
